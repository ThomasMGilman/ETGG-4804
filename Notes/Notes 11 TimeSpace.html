Time Space
  • Explore tradeoff of time for space
Example
  • Suppose we have an input file (or files) with a large amount of text
  • We want to query for data
  • How can we do this efficiently?
Search
  • Brute force: Just scan the string:
def search(needle,haystack):
    for h in range(len(haystack)):
        for n in range(needle):
            if h+n > len(haystack):
                return -1
            if haystack[h+n] != needle[n]:
                break
        else:
            return h #index of match
    return -1
Time
  • How much time does brute force take?
    • O(|H|,|N|)
  • If haystack is large, this could be significant!
  • Crafted data can result in long search times
    • H = ‚ÄúAAAAAAAAAAAAAAAAAAAAAAAAB‚Äù
    • N = ‚ÄúAAAAAAAAAAAAAX‚Äù
Tries
  • Smarter data structure family: A trie
    • From re trie val
  • A tree, but each edge is labelled with a letter (or string)
Suffix Tree
  • We store a trie with entries for every suffix of some input
    • $ = symbol for ‚Äúend of string‚Äù
  • Example: ceded
Prefixes
  • We can combine several single-letter edges into a single edge
Note
  • We can also put labels on nodes
    • Node label = label of edge that ends in that node
Storage
  • Problem: Previous scheme requires O(n¬≤) space
  • We can reduce storage: Use indices of start, end characters
    in the input string
Search
  • Given haystack H: Preprocess to get suffix tree
  • Then: Search for needle N:
    • Start at root of tree
    • Walk through N, choosing tree edges accordingly
    • Stop when
      • Match found
      • Can‚Äôt go further (Report: No match)
  • Time: O(|N|)
  • Haystack can be arbitrarily large!
  • Size of H does not affect time of searches * Only preprocessing time

Issue
  • Usually want to know where match is (not just if it exists)
  • We can tell that easily if we use the index location representation
    • Walk through tree until all characters of needle are consumed
    • If we cannot consume all characters of needle, then it doesn‚Äôt exist
    • Let N be the final node we end up at
    • Find all leaves under N
    • Record total length of substrings encountered as you go to each leaf
    • When at leaf: Report leaf.start - totalLength

Problem

    • But construction is easier

Time Analysis
  • Searching for word w from text T:
    • Suffix tree: Search time = O(w)
    • Suffix array: O(lg T + m)
      • m=number of matches
  • Not an exact bound: Also depends on w (time to find common prefixes in array)
  • When are these equal? Suppose w = 8 characters
    • Then O(lg T) is larger when T=256 characters
    • But in practical terms, maybe not a big deal‚Ä¶
  • Suppose T=2GB
    • O(lg T) = around 31 array accesses
    • There‚Äôs another issue that arises‚Ä¶
Cache
  • Recall: On modern CPU‚Äôs cache effects can be significant
    • Trees = hopping all over memory
    • Cache controller has difficulty predicting
  • So suffix arrays might actually be faster in many cases!
Disk
  • Can keep arrays on disk and selectively load pieces
  • Trees can also be stored on disk, but takes more cleverness to use
  • And we can use interpolation search instead of binary search
  • Interpolation search is O(lg lg n)
    • This is much better than O(lg n)

    L=0

        mid = L + (R-L)/2   #avoid overflow

Interpolation Search
  • Interpolation search doesn‚Äôt always go to middle point
  • Instead, determine where we are relative to universe of keys
  • Ex: Suppose we know H (haystack) consists of English words
  • If N starts with ‚ÄúE‚Äù: We start off with mid = 5/26 = 0.19, not 0.5
Interpolation Search
  • Formula:
  • Let lo = H[L][0]
  • Let hi = H[R-1][0]
  • Let n = N[0]
  • mid = L + (n-lo) / (hi-lo) * (R-L)
  • Then use mid in the previous binary search code
  • Not as good if key distribution is nonuniform (like English language)
